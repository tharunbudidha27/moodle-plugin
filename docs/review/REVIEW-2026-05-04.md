# Code Review Report — `local_fastpix`

Reviewer perspective: Senior Backend Engineer / Moodle plugin reviewer.
Scope reviewed: full plugin source under `local/fastpix/` (excluding `.claude/` tooling, `.git/`, vendored `php-jwt`).
Moodle target: 4.5 LTS. Plugin maturity declared: `MATURITY_BETA`.

---

## 1. Overall Assessment

The plugin shows strong architectural intent. The 3-layer separation (endpoint → service → gateway), idempotent webhook ledger, MUC-backed circuit breaker, RS256-only JWT signing, dual-secret rotation, and SSRF guard are correctly placed. Naming follows frankenstyle, capabilities are minimal, the privacy provider is fully implemented, and scheduled tasks are split correctly between `scheduled_task` and `adhoc_task`. The `.claude/` rule set is unusually disciplined and the source largely respects it.

That said, **the plugin is not installable in its current state**. `db/services.php` and `db/tasks.php` register classes that do not exist in `classes/external/` and `classes/task/` respectively, and the lang file is missing strings that admin and webservice registration paths require. There are also two correctness-grade bugs (CRC32 cache keys, IPv4-only SSRF resolution) that would cause silent wrong-asset reads and SSRF bypasses in production. Several smaller issues (race in salt bootstrap, ledger/task-enqueue lacking a transaction, breaker key collision, no README) are individually minor but compound.

**Verdict: Promising architecture, not production-ready. ~1–2 weeks of focused work to ship.**

---

## 2. Architecture Review

### Strengths
- **3-layer rule** is observed: `webhook.php`, `settings.php`, `lib.php` are thin; services own business rules; `classes/api/gateway.php` is the only place that touches `api.fastpix.io`.
- **Single HTTP boundary**: only `gateway.php` constructs `\core\http_client`. No `curl_*`, no raw Guzzle, no `file_get_contents` against URLs.
- **Idempotency**: webhook table has `UNIQUE(provider_event_id)`; insert path catches `dml_write_exception` and returns 200. Asset insert catches a UNIQUE race in `get_by_fastpix_id_or_fetch`.
- **Concurrency**: `projector` acquires a `\core\lock` keyed `asset_<fastpix_id>`, releases in `finally`, invalidates cache inside the lock. Total ordering uses lex tiebreak via `strcmp`.
- **Circuit breaker** state lives in MUC (`circuit_breaker` cache area) — survives across FPM workers.
- **JWT signing** is local RS256 via vendored `firebase/php-jwt`; no remote `createToken` call.
- **Capability surface** is exactly one (`local/fastpix:configurecredentials`), matches the documented design.
- **Privacy provider** implements all three required interfaces (`metadata::provider`, `request\plugin\provider`, `core_userlist_provider`) and wires GDPR pending + soft-delete correctly.

### Weaknesses
- **Hot-path is implicit**: `get_media` is the only PROFILE_HOT method, but the profile selection lives inline. New methods can silently land on `PROFILE_HOT`. A small enum/method-table would prevent drift.
- **Service constructors are inconsistent**: `gateway`, `verifier`, `credential_service`, `feature_flag_service`, `rate_limiter_service`, `upload_service` all use a custom `instance()` singleton. `asset_service` is **all static**. `jwt_signing_service` has neither `instance()` nor static — callers must instantiate. This inconsistency makes mocking, DI, and testing harder than necessary.
- **`process_webhook` projector instantiation** is direct (`new \local_fastpix\webhook\projector()`), not service-locator. Fine, but inconsistent with the rest of the plugin.
- **No `classes/event/`** despite this being a webhook-driven domain — emitting Moodle events on `media.ready`, `media.failed`, etc., would let `mod_fastpix` and `filter_fastpix` consume the projection without polling.
- **DTOs underused**: `classes/dto/playback_payload.php` exists but I see no other DTOs and no consumer in the read paths. Either commit to DTOs or remove the unused one.
- **Hook directory holds one near-empty class** (`after_config_callback`). Acceptable, but the file is 16 lines — verify it actually does something or delete it.

## 4. Code Quality Issues

### Naming
- `set_config('signing_private_key_previous', …)` co-exists with `webhook_secret_previous` / `webhook_secret_rotated_at` — different naming conventions for the same pattern (rotated key + rotated_at). Pick one shape.
- `feature_flag_service::watermark_enabled()` and `tracking_enabled()` read configs (`feature_watermark_enabled`, `feature_tracking_enabled`) that are **never written** by `db/install.php` or `settings.php` — dead code, or settings are missing.
- `gateway::endpoint_key()` and `asset_service::cache_key_*()` both rely on `crc32b`. Function name doesn't hint at the fact that this is hash-collidable (see §5).

### Readability
- `gateway::request()` is 80 lines with nested `try/catch` inside a `while`. The `gateway_not_found` / `gateway_unavailable` re-throws inside the catch are subtle — a generic `\Throwable` catch above them would silently retry domain exceptions if their order shifted. Consider an explicit fast-fail list or split helpers.
- `decode_body()` returns `(object)['data' => $decoded]` when the body is a JSON array. Magic. Document it or accept arrays explicitly upstream.
- `process_webhook::execute` computes latency as `(microtime(true) * 1000) - ((int)$row->received_at * 1000)` — mixes ms-since-epoch with seconds-then-multiplied. The two clocks come from different sources (request time vs DB-stored received_at). Result is mostly fine but slightly off-by-rounding; use one source.

### Comments
- Several `// rule W7` style references make the code self-auditable. Good.
- The `lib.php` "legacy migration" comment is fine but the file is otherwise empty — consider removing.

### Magic constants
- TTLs (`SESSION_TTL_SECONDS = 86400`, `DEDUP_TTL_SECONDS = 60`, `RETENTION_SECONDS = 7776000`, `1209600`, `300`, `1800`) are scattered as private constants in the relevant classes. Acceptable; just confirm they all match the architecture doc's documented values.

### Error handling
- `upload_service::owner_hash()` falls back to generating a salt if config is empty and persists it. This is racey: two concurrent requests after a fresh install both see empty salt, both generate different salts, the second `set_config` wins and the first request's hash is now orphaned. Bootstrap should be `db/install.php` only (it already is, line 22), so the fallback should `throw` instead of silently re-bootstrapping.
- `credential_service::ensure_signing_key()` doesn't lock, so two simultaneous first-uses can both call `gateway::create_signing_key()` and the second `set_config` wins — leaking a signing key on the FastPix side. Use `\core\lock` or a named-lock pattern.

## 5. Security Concerns

### S-1 (Critical) — CRC32 cache keys are collidable
`asset_service::cache_key_fastpix()` and `cache_key_playback()` use `hash('crc32b', …)` (32-bit / 8-hex output). With ~77,000 assets the birthday-paradox probability of a collision exceeds 50 %. A collision means two distinct `fastpix_id`s map to the same MUC key, so:
- Cache lookup for asset B returns asset A's row → `access_policy`, `drm_required`, `playback_id`, `owner_userid` are all swapped.
- A privileged read could leak metadata across users.
- A DRM-required asset could be served as `private`, breaking the DRM gate at the edge.

**Use** `substr(hash('sha256', $id), 0, 32)` or full sha256 (still alphanumeric, satisfies `simplekeys=true`). Same fix for `gateway::endpoint_key`, `upload_service::dedup_key`, and `projector::cache_key_*`.

### S-2 (High) — SSRF guard is IPv4-only
`upload_service::assert_ssrf_safe()` resolves with `gethostbynamel()`, which returns **only IPv4 A records**. AAAA-only hosts bypass the check entirely (the array is empty → `unresolvable` is thrown — actually that fails closed, OK), but **dual-stack hosts that present a private IPv6** like `fd00::1`, `::1`, `fe80::…` will be passed through if any IPv4 is public. Also, AWS metadata is reachable over IPv6 in some networks.

Switch to `dns_get_record($host, DNS_A | DNS_AAAA)` and validate every IP, including `::1`, `fc00::/7` (ULA), `fe80::/10` (link-local).

Additional issue: this is a **TOCTOU**. The host is resolved once, the gateway then makes a separate call where the resolver re-runs and may get a different answer (DNS rebinding). The gateway-side request needs to either pin the IP (`CURLOPT_RESOLVE` equivalent through Guzzle) or run inside a trusted egress proxy. At minimum, document the gap.

### S-3 (Medium) — `webhook.php` ledger insert + task enqueue is not atomic
If `\core\task\manager::queue_adhoc_task()` fails after the ledger row is inserted, the event sits in `status='pending'` forever; nothing reschedules it. Either wrap in a DB transaction (and let it fail closed → 500 → FastPix retries → idempotent insert wins on retry), or add a `prune_pending_webhook_events` sweeper that re-enqueues `pending` rows older than N minutes.

### S-4 (Medium) — Per-IP rate limiter trusts `getremoteaddr()`
Behind a misconfigured reverse proxy, `getremoteaddr()` can return the proxy IP for all requests — collapsing the rate-limit space. Document the operational requirement that the Moodle reverse-proxy headers (`reverseproxy`, `forwardedaddrs`) are correctly configured. Also consider a global cap, since per-IP alone is bypassable from a botnet.

### S-5 (Low) — `error_log` in `gateway::log_call` and `credential_service`
Functional, but `error_log` writes to whatever PHP-level sink is configured, sidestepping Moodle's logging stack. The redaction-canary tests depend on knowing where logs go. Wrap in a small helper (e.g., `\local_fastpix\util\structured_logger`) and route through `mtrace` / Moodle log stores so `S2` is testable.

### S-6 (Low) — Webhook secret length check
`verifier::verify` checks `strlen(...) < 1`. A short non-empty secret (e.g., 8 chars from a misconfigured admin) passes. Enforce a minimum length (32 hex / 16 bytes) at config-set time and at verify time.

### S-7 (Low) — Signature header format
`hash_hmac('sha256', …)` returns 64-char lowercase hex. If FastPix ships the signature as `sha256=…`, base64, or uppercase, `hash_equals` returns false 100 % of the time. Confirm with the FastPix docs and add a parser if needed (still constant-time). Document the expected exact format in the verifier.

### S-8 (Informational) — Plaintext API secret at rest
`admin_setting_configpasswordunmask` masks the UI; the value is plaintext in `mdl_config_plugins`. This is documented (`S8`), but a `README.md` actually has to exist for that documentation to land.

## 6. Performance & Scalability

- **Static-acceleration cache** on the `asset` MUC area is fine, but `staticaccelerationsize=100` is very small — under heavy course load the LRU thrashes. Bench and tune.
- **`asset_cleanup`** batches at 200, which is sane, but it issues one `gateway::delete_media` per row sequentially. For large GDPR purges this is slow and serializes against retry/breaker. Either parallelise (`waitUntil`-style — N/A on Moodle, so chunked async via adhoc tasks) or document the throughput limit.
- **`webhook_event_pruner`** issues a `count_records_select` then `delete_records_select` — racey between the two. Just delete and `mtrace` the affected count via the `delete` return value or a single SQL `DELETE … RETURNING count`. Minor.
- **`gateway::request` retries with `usleep`** which blocks the worker. For an FPM pool this means up to (200+400+800)ms = 1.4 s of blocked thread per call. Document and consider capping retries when the breaker is half-open.
- **`feature_flag_service` reads `get_config` 2× on every `drm_enabled` call** — fine because Moodle caches plugin config in a request-scoped array, but if you ever call this in a hot loop, fetch once.
- **`projector::insert_from_created_event`** does an unnecessary `update_record` after the insert (insert returns the ID, update sets `last_event_*` again). Fold into a single insert with the event metadata pre-set.
- **Cache TTL of 60 s** on the asset cache caps hit ratio. A `media.ready` projection invalidates the cache, so longer TTLs (e.g., 1 hour) are safe; consider raising it.
- **No bulk fetch API** on `asset_service` — `mod_fastpix` rendering a course page with 20 embedded videos will issue 20 `get_by_fastpix_id` calls. Add `get_many_by_fastpix_id(array $ids)` with a single `get_records_list` and cache batch hydration.

## 7. Moodle Best Practices Violations

| Rule | Violation |
|---|---|
| **M3** | `db/services.php` declares `'capabilities' => 'mod/fastpix:uploadmedia'` — that capability lives in `mod_fastpix`. `local_fastpix` is supposed to consume it (per architecture). This is allowed by the plugin's own `M3` carve-out, but **the plugin will fail to install if `mod_fastpix` is not present**. The `version.php` does NOT declare a dependency, and per `A4` it can't. Net effect: ordering matters at install time, and `db/services.php` registration will explode if the cap doesn't exist. Either (a) gate the registration in `db/services.php` (which Moodle doesn't really support cleanly), (b) rename the cap into `local/fastpix:uploadmedia` and have `mod_fastpix` reference it, or (c) document that the install order is fixed. |
| **M4** | `mtrace` strings throughout tasks are English literals. Moodle's `mtrace` is exempt from translation in practice, but several user-visible places (`get_string('credentials_missing', …, '', 'apikey or apisecret not configured')`) inject English fragments as `$a`. Move to placeholder-only strings. |
| **M5** | `version.php` uses `(int)date('Ymd') . '02'` — dynamic version bumps **are forbidden** by Moodle Plugin standards. Version must be a literal `YYYYMMDDXX` integer that increases monotonically with each release. Replace with `2026050102` (or current). |
| **M6** | The `tests/` folder has unit tests for services, but I see **no `tests/behat/`** scenarios despite settings/admin flows. The doc references behat coverage. |
| **M11** | Cannot verify external API parameter validation — `classes/external/` is missing entirely (F-1). |
| **M12** | `classes/vendor/php-jwt/` is correctly vendored with a `VENDOR.md`. Re-confirm SHA256s before each release. |

10. **`db/log.php`** — if structured logs are intended to flow to Moodle log store.
11. **`feature_watermark_enabled` / `feature_tracking_enabled` settings** — read but never written.
12. **Behat scenarios** for upload, playback, and admin-credential-save flows.
13. **Schema completeness**: `local_fastpix_track` has no indexes despite being foreign-key joined on `asset_id`. Add `idx_asset` on `asset_id`.
14. **`local_fastpix_webhook_event.payload` typed `text`** — fine on most DBs, but on MySQL with default config this is 64 KiB, just above the 1 MiB cap enforced by `webhook.php`. Use `LONG` or document.

## 9. Refactoring Plan (Step-by-Step)

1. Create `classes/external/create_upload_session.php`, `create_url_pull_session.php`, `get_upload_status.php` extending `\core_external\external_api`. Each calls `validate_parameters`, `require_sesskey()` (POST writes), delegates to `upload_service`. Match the contracts in `db/services.php`.
2. Create `classes/task/retry_gdpr_delete.php` extending `\core\task\scheduled_task`. Iterates rows where `gdpr_delete_pending_at IS NOT NULL AND deleted_at IS NOT NULL` and re-invokes `gateway::delete_media`, time-boxed and batched.
3. Add the missing lang strings.
4. Replace dynamic `version.php` with a literal int.
5. Decide and implement install-order strategy for `mod/fastpix:uploadmedia` (recommendation: move the cap into `local_fastpix` and have `mod_fastpix` consume it; reverses A4 ownership but matches dependency direction).

### Phase 1 — Fix correctness bugs (~1 day)
6. Replace `crc32b` with `substr(hash('sha256', $x), 0, 40)` (or full sha256) in: `asset_service::cache_key_*`, `projector::cache_key_*`, `gateway::endpoint_key`, `upload_service::dedup_key`. Add a unit test that hashes 100,000 synthetic IDs and asserts no collisions.
7. Fix SSRF: switch to `dns_get_record($host, DNS_A | DNS_AAAA)`, validate IPv6 ranges (`::1`, `fc00::/7`, `fe80::/10`, `64:ff9b::/96` IPv4-mapped). Add explicit test cases per range.
8. Move salt + signing-key bootstrap into install/upgrade only; remove the silent in-request fallback in `upload_service::owner_hash`. Add a `\core\lock` around `credential_service::ensure_signing_key`.
9. Wrap webhook ledger insert + task enqueue in `\core\transaction` (or accept the existing semantics and add a sweeper for `status='pending'` older than 5 min).

### Phase 2 — Consistency & ergonomics (~3 days)
10. Promote one canonical service-locator pattern. Pick `instance()` singletons (matches the bulk of the codebase) and convert `asset_service` from static to instance. Provide a `set_for_test` seam on each.
11. Centralize cache-key helpers in `\local_fastpix\util\cache_keys`. Have `projector` call `asset_service::invalidate_cache_for(...)` rather than duplicating logic.
12. Introduce `\local_fastpix\util\structured_logger` and route `gateway::log_call`, `credential_service`, projector `debugging` calls through it. Hook redaction.
13. Add `classes/event/` for the documented domain events.
14. Add `asset_service::get_many_by_fastpix_id(array $ids)` with batched DB read + per-key MUC hydration.

### Phase 3 — Hardening (~3 days)
15. Add minimum-length validation on webhook secret at `set_config` time.
16. Document expected `FastPix-Signature` header format; add a parser if non-hex.
17. Tune `asset` MUC `staticaccelerationsize` and `ttl` based on benchmark.
18. Add behat scenarios for the three webservice flows.
19. Write `README.md` with install order, webhook setup, secret-at-rest disclosure, supported versions.
20. Add `idx_asset` index to `local_fastpix_track`.

### Phase 4 — Productionization (ongoing)
21. CI: wire the `.claude/ci-checks/*.sh` into a real pre-commit + GitHub Actions pipeline; today they exist as docs.
22. Add a `phpcs.xml` extending the Moodle ruleset and run `moodle-plugin-ci` end-to-end.
23. Track Moodle 4.5 LTS deprecation deltas (e.g., `\core\http_client` API drift) in CHANGELOG.

## 10. Final Verdict

**Production readiness: NO.**

Reasons:
1. **Two `db/` files reference classes that do not exist** (`classes/external/*`, `classes/task/retry_gdpr_delete`). The plugin will fatal during install or first webservice call.
2. **CRC32 cache keys are collidable** at modest scale and cross-asset metadata leaks are realistic — this is a correctness and security bug rolled into one.
3. **SSRF guard misses IPv6**, defeating the rule it claims to enforce.
4. **Lang strings are incomplete**, install order is undocumented, and there is no `README.md`.

What's good — and there is plenty — is the architecture, the discipline of the rule set, the layering, the projector locking, the JWT design, and the privacy provider. Once the four blocking items are resolved and the §5 correctness bugs fixed, this is a credible 1.0.

**Recommended path:** complete Phase 0 + Phase 1 above, downgrade `MATURITY_BETA` → `MATURITY_ALPHA` until those land, then re-review.
