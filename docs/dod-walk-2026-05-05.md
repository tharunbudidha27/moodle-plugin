## Cleanup (2 items)

### 24. Soft-deleted asset > 7 days hard-deleted; cascade to track table
**DEFERRED.** Current `asset_cleanup` task uses a 90-day GDPR-retention
window (`gdpr_delete_pending_at < (time() - 7776000)`), not a 7-day
soft-delete purge. The architecture spec calls for a separate
`purge_soft_deleted_assets` task at 7d retention; this was renamed to
`asset_cleanup` during Tier 1 with different semantics. The 7-day soft
purge is **missing**. Action: either add a separate task or document
the architecture change.

### 25. Soft-deleted boundary: 6d 23h NOT purged; 7d 1m IS purged
**DEFERRED.** Same as item 24 — the 7-day window doesn't exist. Once
item 24 is resolved, this becomes a unit test on the boundary.

## Upload dedup + SSRF (3 items)

### 29. Two upload calls within 60s for same triple return identical session_id, second has deduped=true
**PASS.** `test_create_file_upload_session_within_60s_returns_deduped_true`
— gateway mocked with `expects($this->once())` so the second call MUST

### 30. Dedup boundary: 59s deduplicates; 61s creates new
**PARTIAL.** `test_create_file_upload_session_after_60s_creates_new_session`
simulates TTL expiry by purging the dedup cache between calls

### 31. SSRF: localhost, RFC1918, link-local, AWS metadata IP, DNS-rebinding
**PASS.** All listed protections are tested:
- `test_create_url_pull_session_rejects_localhost`
- `test_create_url_pull_session_rejects_rfc1918_ip_directly`
- `test_create_url_pull_session_rejects_loopback_ip_directly`
- `test_create_url_pull_session_rejects_link_local_aws_metadata`
- IPv6 variants added in T1.3
- `test_create_url_pull_session_rejects_credentials_in_url` (C1, 2026-05-06)

**DNS-rebinding empirical reframe (2026-05-06).** The original concern was
that the SSRF guard validates IPs at submission time but a malicious DNS
server could rebind to a private IP at the moment of fetch. Audit on
2026-05-06 found that Moodle never directly fetches `source_url`. The
gateway packs the URL into a JSON body and POSTs it to api.fastpix.io;
FastPix's backend later fetches the URL with FastPix's own resolver.
CURLOPT_RESOLVE pinning on Moodle's cURL handle has zero effect on
FastPix's fetch. The residual rebinding risk is FastPix's to mitigate
on their side. Our guard remains in place as defense in depth and is
documented at `classes/service/upload_service.php::assert_ssrf_safe()`.

## Architecture drift to surface

Two items reveal that the implementation has diverged from the
architecture doc in ways that need an ADR or a doc update, not a
code fix:

1. **Cleanup task semantics** (DoD §24-25). Architecture says 7-day soft
   purge as a separate task; we have a 90-day GDPR retention task. The
   90-day window is correct for GDPR but doesn't replace the 7-day
   purge. Either both tasks exist or the architecture doc updates to
   say there's only one task at 90 days.

2. **GDPR alert threshold** (DoD §34). Architecture says 6 failures →
   Moodle event. T3.5 ships 10 failures → mtrace CRITICAL. The
   thresholds and signal types both differ. Pick one and write it down.
